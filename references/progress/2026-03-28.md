# Progress — 2026-03-28

## Project Goal

Parse LINE group chat exports and automatically extract 曉寒's daily work sessions
(online/offline pairs) into a master timecard CSV.

---

## What Has Been Built

### Core pipeline

| Module | Role |
|--------|------|
| `parser.py` | Parse raw LINE `.txt` exports into `ChatMessage` objects |
| `extractor.py` | **Rule-based** online/offline detection via keyword lists |
| `llm_validator.py` | Gemini classifier for messages that are ambiguous to rules |
| `llm_extractor.py` | **Full LLM** path: sends 曉寒's messages per-day to Gemini, returns sessions as JSON |
| `timecard.py` | Build DataFrame, merge into master CSV, export to Excel |
| `cli.py` | Typer CLI (`parse`, `run`, `merge`, `show`, `export`) |

### Two extraction paths

1. **Rule-based** (`timecard parse` / `timecard run --rule-based`): keyword lists in
   `extractor.py`; optionally sends ambiguous patterns to `llm_validator.py`.
2. **Full LLM** (`timecard run`, default): `llm_extractor.py` batches each day's
   messages and asks Gemini to identify sessions directly, with the previous day
   included as context.

### State / incremental runs

A `.state.json` sidecar next to the master CSV stores `last_processed_date`; subsequent
runs re-process only the delta.

### Tests

Unit tests in `tests/test_extractor.py` and `tests/test_llm_validator.py` cover
keyword rules and the LLM validator against a fixture file.

## Current Issues

### 1. Overlapping / phantom sub-sessions (LLM path) — most critical

Several days in `data/timecard_master.csv` show a session whose `online_time` falls
**inside** an already-reported earlier session on the same day:

| Date | Session | Online | Offline | Note |
|------|---------|--------|---------|------|
| 2023-10-03 | 2 | 17:11 | 23:30 | correct outer session |
| 2023-10-03 | 3 | 21:56 | 23:30 | 21:56 is inside s2 — duplicate |
| 2023-10-19 | 1 | 17:33 | 22:53 | correct outer session |
| 2023-10-19 | 2 | 20:25 | 22:53 | 20:25 is inside s1 — duplicate |
| 2023-11-01 | 1 | 17:07 | 22:59 | correct |
| 2023-11-01 | 2 | 22:06 | 22:59 | 22:06 inside s1 — duplicate |
| 2023-11-09 | 1 | 16:27 | 22:29 | correct |
| 2023-11-09 | 2 | 21:44 | 22:29 | 21:44 inside s1 — duplicate |
| 2023-11-14 | 1 | 16:37 | 22:17 | correct |
| 2023-11-14 | 2 | 18:56 | 22:17 | 18:56 inside s1 — duplicate |

**Root cause**: When she makes a temporary departure ("先下，等等回來") and then
returns ("回來上線"), Gemini sometimes reports the return as a *new* session rather
than treating the departure as part of the same session.  The deduplication key in
`_deduplicate_entries` is `(date, online_time)`, so sessions with distinct online
times are never collapsed — even if one is nested inside another.

Also: `2023-11-05` and `2023-11-06` share identical times `17:39 → 22:03`, which
looks like a Gemini hallucination repeating one day's result onto the next.

```
2023-10-30  s2  23:29  —

```

Possible causes:

- She genuinely didn't post an offline message those sessions (session left open at
  end of day).
- Her farewell used phrasing not in `_OFFLINE_KEYWORDS` (rule-based path) or that
  the LLM prompt didn't classify as an offline signal.
- She said "先下" as part of a **temporary** departure that the LLM classified as
  the session's offline_time, then when she returned the LLM opened a new session
  with no matching offline — leaving the second session open-ended (interplay with
  issue #1).

### 3. Rule-based pairing lacks gap-based session merging

`extractor.py`'s greedy pairing (`online → next offline`) has no concept of a
"temporary departure."  The legacy `generate_timecard.py` (still in the repo) used
a 2-hour gap heuristic: if she returns within 120 minutes of going offline, the
departure counts as a break, not a session split.  The refactored extractor discarded
this logic; temporary returns are now split into separate sessions rather than merged.

### 4. Broad keyword `先下` may produce false positives (rule-based)

`"先下"` in `_OFFLINE_KEYWORDS` is short enough to match phrases like
`"先下來確認"` (come down to check, not log off) which are not in `_OFFLINE_EXCLUSIONS`.
`"先下午"` is excluded, but other combinations are not.

### 5. LLM receives no group-chat context

`LLMExtractor.extract_sessions()` pre-filters to `m.sender == TARGET_SENDER` before
splitting into daily chunks.  Gemini therefore only sees 曉寒's own messages.  When
she posts an ambiguous message like `"先下"` as a short reply, the model cannot use
the surrounding conversation to determine whether she means "logging off" or something
else.

---

## Root Cause Summary

The core offline-time detection failures come from two separate problems:

1. **LLM prompt does not reliably treat temporary departures as part of the same
   session.**  The rule in the system prompt ("暫時離開不算結束") is not specific
   enough; Gemini frequently creates a new session for the return instead of merging
   it with the original.

2. **No post-processing validation** catches sessions whose `online_time` falls
   inside an already-existing session on the same day.

## Next Steps (Proposed)

1. **Add overlap detection** in `_deduplicate_entries` / `_renumber_sessions`:
   after collecting all sessions for a day, check for entries where `online_time`
   falls within an earlier session's `[online_time, offline_time]` range and merge
   or discard them.

2. **Strengthen the LLM system prompt** for temp-departure handling: give concrete
   examples where she leaves briefly and returns within the same session (include the
   actual gap threshold, e.g., "≤ 2 hours = same session").

3. **Include a small window of other senders' messages** around each departure event
   for context, rather than filtering to 曉寒-only.

4. **Add false-positive guard for `先下`** in `_OFFLINE_EXCLUSIONS` (e.g.,
   `先下來`, `先下車`) or narrow the pattern to `先下線|先下囉|先下喔|先下了`.

5. **Manual audit of sessions with no offline_time** against the raw chat log to
   determine whether the missing times are true gaps or missed detections.
