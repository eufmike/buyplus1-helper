# Progress — 2026-03-28

## Project Goal

Parse LINE group chat exports and automatically extract 曉寒's daily work sessions
(online/offline pairs) into a master timecard CSV.

---

## What Has Been Built

### Core pipeline

| Module | Role |
|--------|------|
| `parser.py` | Parse raw LINE `.txt` exports into `ChatMessage` objects |
| `extractor.py` | **Rule-based** online/offline detection via keyword lists |
| `llm_validator.py` | Gemini classifier for messages that are ambiguous to rules |
| `llm_extractor.py` | **Full LLM** path: sends 曉寒's messages per-day to Gemini, returns sessions as JSON |
| `timecard.py` | Build DataFrame, merge into master CSV, export to Excel |
| `cli.py` | Typer CLI (`parse`, `run`, `merge`, `show`, `export`) |

### Two extraction paths

1. **Rule-based** (`timecard parse` / `timecard run --rule-based`): keyword lists in
   `extractor.py`; optionally sends ambiguous patterns to `llm_validator.py`.
2. **Full LLM** (`timecard run`, default): `llm_extractor.py` batches each day's
   messages and asks Gemini to identify sessions directly.

### Taiwan Time as the processing standard

The LINE export uses the device's Pacific Time clock.  Before any grouping or LLM
processing, all of 曉寒's messages are converted to **Taiwan Time (UTC+8)** via
`_messages_to_tw()`.

- PDT (UTC-7, Mar–Nov): PT + 15 h → TW
- PST (UTC-8, Nov–Mar): PT + 16 h → TW

Her typical evening shift starts at ~17:00 PT, which is ~08:00–09:00 TW the
following calendar day.  Processing in TW ensures the online and offline events of
the same shift land on the same TW calendar day, eliminating midnight-crossing
splits.

All times stored in the master CSV are **Taiwan Time**.  `online_time_gmt` and
`offline_time_gmt` columns are derived from those (TW − 8 h) for reference.

### State / incremental runs

A `.state.json` sidecar next to the master CSV stores `last_processed_date`;
subsequent runs re-process only the delta.

### Tests

Unit tests in `tests/test_timecard.py`, `tests/test_llm_extractor.py`, and
`tests/test_llm_validator.py` cover the TW conversion, keyword rules, and the LLM
validator.

## Current Issues

### 1. Missing offline times

31 rows have an empty `offline_time`.  At 4.3% this is well below the pre-TW-fix
baseline (6.6%) and the earlier contaminated run (10.3%).

Estimated breakdown:

- **Genuine no-announcement (~60–70%)**: she stopped responding without a farewell.
- **Phrasing not covered by the prompt (~20–30%)**: informal goodbyes the LLM did
  not classify as logout signals.
- **Late-night / short sessions (~10%)**: sessions that start late (TW ~00:00–06:00)
  and end with no explicit sign-off.

### 2. Rule-based pairing lacks gap-based session merging (legacy path)

`extractor.py`'s greedy pairing (`online → next offline`) has no concept of a
"temporary departure."  The legacy `generate_timecard.py` used a 2-hour gap
heuristic; the refactored extractor discarded this.  Affects only the `--rule-based`
path; the default LLM path handles temp departures natively via the system prompt.

### 3. Broad keyword `先下` may produce false positives (rule-based)

`"先下"` in `_OFFLINE_KEYWORDS` is short enough to match phrases like
`"先下來確認"` (come down to check, not log off).  Affects only the rule-based path.

### 4. LLM receives no group-chat context

`extract_sessions()` pre-filters to `m.sender == TARGET_SENDER` before splitting
into daily chunks.  Gemini only sees 曉寒's own messages, which may cause ambiguous
short messages to be misclassified.

## What Was Fixed in This Session

### Midnight-crossing session splits (root cause of missing offline in Feb 2026)

**Problem**: 5 of the original 10 missing-offline sessions in Feb 2026 were caused
by her logging on at ~22:xx PT and logging off at ~00:xx PT the next calendar day.
The PT-day boundary split the session: the offline message appeared on the *next* PT
day's chunk, leaving the current day's session open-ended.

**Fix**: Convert all messages from PT to TW before chunking (`_messages_to_tw()`).
22:06 PST Feb 7 → 14:06 TW Feb 8; 00:34 PST Feb 8 → 16:34 TW Feb 8 — same TW day,
no split.

### Rule 6 removed (phantom sessions)

An earlier prompt included rule 6 — "if no explicit 上線 but there is a 下線, use
the first message of the day as start."  When PT→TW conversion was introduced, some
TW calendar days ended up with only 1–2 messages (spillover from the conversion).
Rule 6 fired on these thin days and created phantom sessions, inflating the row count
from the expected ~714 to 846 and pushing the missing offline rate up to 10.3%.
Removing rule 6 eliminated the phantom sessions.
